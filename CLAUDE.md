## Architecture decisions (don't re-litigate)

- **Vanilla JS ES modules, no bundler.** The original site was static HTML;
  preserving that means zero build step and zero deploy complexity. Imports
  go through esm.sh CDN. If you're tempted to add Vite/Next, ask first.
- **Supabase + RLS, no custom backend.** Anon key + RLS handles read menu /
  write pending reservations / write pending orders. Service role only inside
  Vercel functions for status mutations the customer can't do themselves
  (payment confirmation, marking orders ready, etc).
- **Cart in localStorage**, not in Supabase. The cart is ephemeral browser
  state; only the placed order is persisted. Cross-tab sync via the `storage`
  event in `js/cart.js`.
- **Payments behind a single abstraction** (`js/payments.js` → `requestPayment`).
  Mock mode shows a DEMO-MODE labelled UPI overlay; the Razorpay branch is
  fully written and switches on the `PAYMENT_MODE` flag + env vars.
- **One owner**, no roles. The admin console is single-user. If multi-staff
  becomes a need later, add a `staff_roles` table + RLS, don't bolt onto auth.

## Database schema (live in Supabase)

Migrations in `supabase/migrations/`:

- `0001_init_schema.sql` — `menu_categories`, `menu_items`, `reservations`, `orders`
  with enums (`reservation_status`, `order_type`, `payment_status`, `order_status`),
  indexes, `updated_at` triggers.
- `0002_rls_policies.sql` — anon: read active menu + insert pending reservations
  + insert placed/unpaid orders; authenticated: everything.

Seed: `supabase/seed/menu_seed.sql`, regenerated by `python3 scripts/extract_menu.py`
from `menu.html` (legacy static markup is preserved for the extractor only —
the live menu page renders from Supabase).

## First-run sanity tests (do these on day one in Claude Code)

1. **Static server up**: `python3 -m http.server 8080` → open http://localhost:8080
   → home, about, gallery, contact all render with their original design intact.

2. **Menu reads from Supabase**: open /menu.html → should see 30 sections
   populated with items, sticky category jumpbar at top, Add buttons on each
   row, floating cart FAB bottom-right. If you see "Loading the menu…" forever,
   it's a CORS or network issue — check the browser console.

3. **Cart works**: tap +, see toast, FAB count goes up, open drawer, qty
   steppers work, Remove works. Drawer persists across page reloads
   (localStorage).

4. **Reservation submit**: open /reservation.html → fill the form → submit.
   Should see the confirmation card with a real reference ID. Confirm the
   row appears in Supabase Studio: `select * from reservations`.

5. **Pre-order pickup**: menu → add 2-3 items → cart drawer → "Continue to
   Pre-order" → review → keep mode "Pickup" → fill form → submit. Should
   land on confirmation with an order id. Confirm in Supabase:
   `select * from orders where type = 'pickup'`.

6. **Pre-order with table**: same flow, switch mode to "With your Table" →
   fill all the extra fields → "Continue to UPI" → the DEMO MODE overlay
   appears → "Confirm Payment" → confirmation. Without Vercel functions
   running, the mock payment endpoint fails and you'll see a note about
   the order sitting in pending — that's expected for local-only. With
   `vercel dev` running, the endpoint resolves and the order is marked
   `mock_paid`.

7. **Admin login**: /admin.html → owner@banjara.in / Banjara@2026 →
   Menu tab → edit a price, watch the inline Save button appear → save →
   refresh /menu.html in another tab → price updates within 5 min (cache TTL).
   Then check Reservations tab and Orders tab — your test rows from steps
   4-6 should be there.

  point of the architecture is zero-build static deploy.

  may contain public values. Secrets go in Vercel env + `.env.local`.
- Don't bypass RLS for convenience. If a client-side write fails because
  of RLS, that's RLS doing its job — write a serverless function instead.
- Don't refactor the design system. The Playfair / Cinzel / Cormorant +
  gold-on-deep-black aesthetic is core to the THEKULTHOUSE brand promise
